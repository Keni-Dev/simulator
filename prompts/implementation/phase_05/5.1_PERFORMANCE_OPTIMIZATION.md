# Phase 5 — Step 5.1: Performance Optimization

## Context
With all features built and integrated, this step profiles the application and optimizes performance bottlenecks. The target is stable 30-60 FPS on a MacBook Air M1 during maximum particle emission with hand tracking active. The three main performance concerns are: (1) MediaPipe hand tracking CPU cost, (2) Three.js particle rendering, and (3) React re-render frequency from rapid state updates.

## Skills to Load
- `web-performance-optimization` — Core Web Vitals, bundle analysis, runtime profiling, memory leak detection
- `3d-web-experience` — Three.js-specific optimizations: geometry instancing, draw call reduction, texture management, LOD
- `systematic-debugging` — Systematic approach to identifying root causes of performance issues

## Prerequisites
- Phases 0-4 complete: Full application functional with hand tracking, physics, VFX, HUD, and material selector
- Access to browser DevTools Performance tab and Chrome Task Manager

## Instructions

### Step 5.1.1: Create Performance Utility Module
**File:** `src/utils/performance.ts` — CREATE
**Purpose:** Utility functions for FPS monitoring, frame time tracking, and performance logging
**Details:**
- Create a `PerformanceMonitor` class (singleton):
  - Tracks FPS over a rolling window of 60 frames
  - Methods: `startFrame()`, `endFrame()`, `getFPS()`, `getAverageFrameTime()`
  - Optional: log warnings when FPS drops below 30 for more than 2 seconds
- Create function `createThrottledCallback(callback: Function, targetFPS: number)`:
  - Returns a function that wraps the callback and only executes it at most `targetFPS` times per second
  - Uses `performance.now()` for accurate timing
  - This is used to throttle MediaPipe processing to 30fps while rendering at 60fps

### Step 5.1.2: Optimize MediaPipe Processing
**File:** `src/features/hand-tracking/hooks/use-hand-tracking.ts` — MODIFY
**Purpose:** Ensure hand tracking runs at 30fps max to leave CPU headroom for rendering
**Details:**
- Use `createThrottledCallback` to limit MediaPipe `processFrame` calls to 30fps
- Move MediaPipe processing to a Web Worker if possible:
  - Create a worker file that receives video frame data (via `OffscreenCanvas` or `ImageBitmap`)
  - Processes landmarks in the worker thread
  - Posts results back to the main thread
  - CAVEAT: `@mediapipe/tasks-vision` may not support Web Workers due to WASM/WebGL dependencies. If workers don't work, keep on main thread with throttling
- Reduce `modelComplexity` to 0 (fastest mode) if FPS is consistently below 30
- Only process frames when hands are likely to be present — skip frames when no hand has been detected for the last 30 frames (reduce to 10fps idle processing, back to 30fps on detection)

### Step 5.1.3: Optimize Particle System
**File:** `src/features/simulation/components/ElectronParticles.tsx` — MODIFY
**Purpose:** Ensure particle system uses object pooling and minimal draw calls
**Details:**
- Implement a proper particle pool:
  - Pre-allocate a `Float32Array` for positions, sizes, and opacities for MAX_PARTICLES (100)
  - Use `BufferAttribute` and `needsUpdate = true` to update positions each frame
  - Do NOT create new `THREE.Vector3` objects or push/splice arrays each frame — reuse pre-allocated objects
- Use a single `Points` object with custom `ShaderMaterial` or `PointsMaterial`:
  - One draw call for ALL particles (instead of one mesh per particle)
  - Custom vertex shader handles per-particle size, opacity, and color
- Cap active particles dynamically:
  - If FPS < 30: reduce max particles to 30
  - If FPS < 20: reduce to 10 or disable particles temporarily and show a simpler glow effect
  - When FPS recovers > 40: restore full particle count

### Step 5.1.4: Optimize React Re-renders
**File:** Multiple files — MODIFY
**Purpose:** Prevent unnecessary React re-renders from high-frequency store updates
**Details:**
- Audit all `useSimulationStore` and `useHandTrackingStore` subscriptions:
  - Every subscription MUST use a selector: `useStore(s => s.specificField)`, never `useStore()`
  - Use `shallow` comparison for object selectors: `useStore(s => ({ q1: s.q1, q2: s.q2 }), shallow)`
- HUD charge displays: These update at ~60fps via store. To prevent DOM thrashing:
  - Use `requestAnimationFrame` to batch charge display updates to 30fps max
  - Or use React's `useDeferredValue` for non-critical HUD elements (status bar)
  - Or update DOM directly via refs for the numeric displays (bypass React diffing):
    ```
    // Pattern: ref.current.textContent = formatCharge(charge)
    ```
- Wrap pure components in `React.memo()` — especially MaterialCard, MaterialLabel
- The 3D scene (inside Canvas) already avoids React re-renders since `useFrame` is imperative

### Step 5.1.5: Optimize Three.js Rendering
**File:** `src/features/simulation/components/SimulationCanvas.tsx` — MODIFY
**Purpose:** Canvas-level rendering optimizations
**Details:**
- Set `frameloop="demand"` on Canvas to render only when state changes (if feasible — may not work with `useFrame` loops that need continuous rendering. Use `"always"` if the scene has continuous animation)
- Limit DPR: `dpr={Math.min(window.devicePixelRatio, 1.5)}` instead of 2 on retina displays
- Enable `gl.powerPreference: 'high-performance'` (should already be set from Phase 0)
- Dispose Three.js resources when materials are swapped:
  - When a MaterialMesh unmounts or geometry changes, dispose the old geometry and material to prevent GPU memory leaks
  - Use `geometry.dispose()` and `material.dispose()` in cleanup
- Minimize draw calls:
  - Use `InstancedMesh` for particles if converting from Points to mesh-based particles
  - Merge static geometries if multiple objects share the same material

### Step 5.1.6: Optimize Collision Detection
**File:** `src/features/physics/hooks/use-collision-detection.ts` — MODIFY
**Purpose:** Reduce collision check cost
**Details:**
- Cache the base bounding box geometry and only translate it when position changes (avoid full `setFromObject` every frame)
- Use sphere collision first (broadphase): if the distance between centers > sum of radii, skip AABB check
- Only check collision when both objects are being grabbed and moved (if neither hand is grabbing, objects are stationary and collision state won't change)

## Rollback Plan
- Performance optimizations are incremental — each sub-step can be reverted independently
- If throttling breaks hand tracking: restore the full-speed RAF loop
- If particle pool breaks VFX: revert to simple array-based particles

## Troubleshooting
| Symptom | Likely Cause | Fix |
|---|---|---|
| FPS drops below 20 | Too many particles or MediaPipe on main thread | Reduce MAX_PARTICLES to 30, reduce model complexity to 0 |
| Memory keeps growing | GPU resources not disposed on material swap | Add cleanup: `geometry.dispose()`, `material.dispose()`, texture disposal |
| Janky HUD numbers | DOM updates blocking main thread | Use ref-based DOM updates instead of React state for charge displays |
| Tab crash after 5+ minutes | Memory leak in RAF loop or particle array | Check for arrays that grow unbounded; ensure dead particles are recycled |
| High CPU even when hands not visible | MediaPipe processing at full speed when idle | Reduce framerate to 10fps when no hands detected for 30 frames |
