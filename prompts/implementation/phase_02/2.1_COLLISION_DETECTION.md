# Phase 2 — Step 2.1: Collision Detection System

## Context
The physics engine requires knowing WHEN the two material objects are overlapping (colliding) to calculate charge transfer. This step implements AABB (Axis-Aligned Bounding Box) collision detection between the left and right material meshes. When the user rubs the objects together via hand tracking, the system detects the overlap and calculates the relative velocity of the collision — both critical inputs to the triboelectric math engine (Step 2.2).

## Instructions

### Step 2.1.1: Create Collision Utility Functions
**File:** `src/features/physics/utils/collision.ts` — CREATE
**Purpose:** Pure functions for AABB collision detection and velocity computation
**Details:**
- Create function `checkAABBCollision(boxA: THREE.Box3, boxB: THREE.Box3): boolean`:
  - Returns true if the two axis-aligned bounding boxes overlap in all three axes
  - Use Three.js's built-in `Box3.intersectsBox()` method
- Create function `computeBoundingBox(mesh: THREE.Mesh): THREE.Box3`:
  - Creates a new `THREE.Box3().setFromObject(mesh)` that accounts for the mesh's world transform (position, rotation, scale)
  - Returns the AABB in world space
- Create function `calculateOverlapDepth(boxA: THREE.Box3, boxB: THREE.Box3): number`:
  - Returns the minimum overlap distance across all three axes
  - This can be used to determine "how deep" the collision is, which could scale the charge transfer rate
  - If no overlap, return 0
- Create function `calculateRelativeVelocity(posA: THREE.Vector3, prevPosA: THREE.Vector3, posB: THREE.Vector3, prevPosB: THREE.Vector3, deltaTime: number): number`:
  - Velocity of A: `(posA - prevPosA) / deltaTime`
  - Velocity of B: `(posB - prevPosB) / deltaTime`
  - Relative velocity: magnitude of `velocityA - velocityB`
  - Returns the scalar magnitude in world-units per second
  - This is the $v_{rel}$ term in the triboelectric equation

### Step 2.1.2: Create Collision Detection Hook
**File:** `src/features/physics/hooks/use-collision-detection.ts` — CREATE
**Purpose:** Runs every frame inside the R3F render loop, checks for collisions between the two material meshes, and updates the simulation store
**Details:**
- Accept refs to both material meshes as parameters (or retrieve them from a shared context/registry)
- Use `useFrame((state, delta) => { ... })` to run on every render frame
- On each frame:
  1. Compute bounding boxes for both meshes using `computeBoundingBox()`
  2. Check collision using `checkAABBCollision()`
  3. If colliding:
     - Calculate relative velocity using stored previous positions
     - Update store: `setCollisionState(true, relativeVelocity)`
  4. If NOT colliding:
     - Update store: `setCollisionState(false, 0)`
  5. Store current positions as "previous positions" for the next frame's velocity calculation (use `useRef` to persist between frames)
- PERFORMANCE: Bounding box computation should use `computeBoundingOnce` then translate based on position changes, rather than recomputing the full geometry bounds every frame. Or use a simplified sphere collider (distance between centers < sum of radii) for even faster collision checks
- The collision check runs EVERY frame (~60fps) but is computationally cheap (just 3 axis comparisons for AABB)

## Troubleshooting
| Symptom | Likely Cause | Fix |
|---|---|---|
| Objects never register as colliding | Bounding box not using world transform | Use `Box3.setFromObject(mesh)` which accounts for world matrix, not local geometry |
| Objects collide when far apart | Bounding box too large | Check if the geometry's bounding box includes vertices far from center. Scale it down or use a sphere collider |
| Relative velocity is always 0 | Previous position not being stored correctly | Ensure `useRef` stores the last known position and updates AFTER the velocity calculation |
| Collision detection causes lag | Recomputing geometry bounds every frame | Cache the base bounding box and only translate it by the mesh's position delta |
