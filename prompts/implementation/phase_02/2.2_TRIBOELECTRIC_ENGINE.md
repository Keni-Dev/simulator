# Phase 2 — Step 2.2: Triboelectric Math Engine

## Context
This is the scientific core of the simulator. With collision detection providing `isColliding` and `relativeVelocity` (Step 2.1), this step implements the triboelectric charge transfer equation: $\Delta q = k \cdot v_{rel} \cdot \Delta t \cdot (T_A - T_B)$. The engine runs every frame during a collision, updating `q1` and `q2` in the store while strictly enforcing conservation of charge ($q_{net} = 0$).

## Skills to Load
- `research-engineer` — Ensure the mathematical model is physically rigorous, handles edge cases (zero velocity, same material, conductor grounding), and avoids floating-point drift
- `clean-code` — Pure functions for all math operations, comprehensive edge case handling

- Step 2.1 complete: Collision detection provides `isColliding` and `relativeVelocity` per frame
- Constants defined: `FRICTION_COEFFICIENTS` matrix, material triboelectric indices, `MAX_CHARGE`, `GROUNDING_RATE`

## Instructions

### Step 2.2.1: Create the Triboelectric Engine
**File:** `src/features/physics/utils/triboelectric-engine.ts` — CREATE
**Purpose:** Pure mathematical functions that calculate charge transfer between materials
**Details:**
- Create function `getFrictionCoefficient(materialA: Material, materialB: Material): number`:
  - Looks up the specific `k` value from `FRICTION_COEFFICIENTS` for this material pair
  - The lookup should be symmetric: `k(A, B) === k(B, A)`
  - If the pair is not found (shouldn't happen with MVP materials), default to a fallback k of 0.3
  - If both materials are the same, return 0 (same material rubbing itself produces no charge transfer)

- Create function `calculateChargeTransfer(params: { frictionCoefficient: number, relativeVelocity: number, deltaTime: number, triboIndexA: number, triboIndexB: number }): number`:
  - Implements: $\Delta q = k \cdot v_{rel} \cdot \Delta t \cdot (T_A - T_B)$
  - `k` = friction coefficient (from lookup)
  - `v_rel` = relative velocity (from collision detection)
  - `\Delta t` = time since last frame in seconds
  - `T_A - T_B` = difference in triboelectric indices (determines charge direction: positive means A loses electrons to B)
  - Returns `deltaQ` in μC
  - The sign of deltaQ determines direction: positive deltaQ means material A becomes more positive (loses electrons), material B becomes more negative (gains electrons)
  - IMPORTANT: Normalize the triboelectric index difference. If raw difference is large (e.g., Glass 25 - Rubber -70 = 95), scale it down with a normalization factor so charge transfer rates are visually meaningful. Use a `TRIBO_NORMALIZATION_FACTOR` constant (e.g., 0.001) to keep charge transfer per second in a reasonable range (0-10 μC/s at moderate rubbing speed)

- Create function `applyChargeTransfer(q1: number, q2: number, deltaQ: number, maxCharge: number): { newQ1: number, newQ2: number }`:
  - Applies deltaQ: `newQ1 = q1 + deltaQ`, `newQ2 = q2 - deltaQ`
  - Enforces conservation: `newQ1 + newQ2` MUST equal the previous `q1 + q2` (which should be 0 for insulators)
  - Clamps both charges to ±MAX_CHARGE — if one hit's the cap, don't transfer more charge
  - Applies floating-point precision fix: round both to `CHARGE_PRECISION` decimal places to prevent drift
  - Returns the new charge values

- Create function `applyGroundingEffect(charge: number, groundingRate: number, deltaTime: number): number`:
  - For conductors touching the "hand" (human body = ground), discharge toward 0
  - Implements exponential decay: `newCharge = charge * Math.exp(-groundingRate * deltaTime)`
  - If charge is very close to 0 (within 0.01 μC), snap to exactly 0 to prevent infinite decay
  - Returns the new charge value

- Create function `isConservationMaintained(q1: number, q2: number, tolerance: number): boolean`:
  - Validates that `Math.abs(q1 + q2)` is within `tolerance` (e.g., 0.001 μC)
  - Used as a runtime safety check — if conservation is violated, log a warning
  - For conductors, conservation with the "ground" (human body) is implicit — the ground absorbs the charge

### Step 2.2.2: Create the Physics Loop Hook
**File:** `src/features/physics/hooks/use-physics-loop.ts` — CREATE
**Purpose:** Runs the triboelectric engine every frame, reading collision data and updating charges in the store
**Details:**
- Use `useFrame((state, delta) => { ... })` to run in the R3F render loop
- On each frame:
  1. Read from simulation store: `isColliding`, `relativeVelocity`, `leftMaterial`, `rightMaterial`, `q1`, `q2`
  2. If `isColliding` AND `relativeVelocity > 0` (objects are rubbing):
     a. Get friction coefficient for the current material pair
     b. Calculate `deltaQ` using `calculateChargeTransfer()`
     c. Check if either material is a conductor:
        - If conductor: apply grounding effect — the conductor discharges to the "human body" (ground). Update only the conductor's charge toward 0
        - Special case: when a conductor touches an insulator, the insulator transfers charge TO the conductor, and the conductor immediately grounds it. The net effect: the insulator loses charge, the conductor stays at ~0
     d. If both are insulators: apply normal charge transfer using `applyChargeTransfer()`
     e. Update store: `updateCharges(deltaQ)` for insulators, or `groundCharge()` for conductors
     f. Run conservation check: `isConservationMaintained()`. If violated, log warning to console
  3. If NOT colliding:
     - No charge transfer. Charges remain as they are. Insulators hold their charge indefinitely
     - If debug mode is on, log current charges periodically (every 60 frames to avoid spam)
- Use `getState()` for reading store values inside `useFrame` to avoid stale closures

## Troubleshooting
| Symptom | Likely Cause | Fix |
|---|---|---|
| Charges don't change when rubbing | `isColliding` is never true | Debug collision detection (Step 2.1). Log the bounding boxes and collision state |
| Charges change too fast | `k` or normalization factor too high | Reduce `TRIBO_NORMALIZATION_FACTOR` or individual k values in the friction matrix |
| Charges change too slowly | Normalization too aggressive | Increase the factor. Also check if `relativeVelocity` is returning reasonable values |
| `q1 + q2 ≠ 0` after some rubbing | Floating-point drift | Ensure `applyChargeTransfer` enforces `newQ2 = -(q1 + deltaQ)` rather than `q2 - deltaQ` independently |
| Conductor doesn't ground | Grounding logic not triggered | Check that the conductor material `type === 'conductor'` check is correct and `applyGroundingEffect` is being called |
| Same materials produce charge | Missing same-material check | Verify `getFrictionCoefficient` returns 0 when both materials are identical |
